# 10 — FOSS Component Audit

This document evaluates all open-source candidates for every functional component
in the Open Mailgroupware stack. **Multi-tenancy capability is the primary
evaluation criterion** — every component must work in a shared deployment where
multiple tenants' data is strictly isolated.

## Evaluation matrix

Each component is scored on:

- **Operational simplicity** — ease of deployment, configuration, maintenance
- **Protocol coverage** — how many required protocols it speaks
- **Extensibility** — how easily it integrates with config compiler and product plane
- **Backup story** — how cleanly its state can be backed up and restored
- **Search quality** — native search or integration with external search
- **Groupware maturity** — calendar, contacts, shared resources, delegation
- **Abuse integration** — how well it integrates with Rspamd/ClamAV pipeline
- **Multi-tenancy** — native tenant isolation vs. workaround-needed

## 1. SMTP Ingress / MTA

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| Postfix | C | IPL | 3.10+ | 25+ years |
| Stalwart SMTP | Rust | AGPL 3.0 | 0.22+ | Yes |
| Haraka | Node.js | MIT | 3.0+ | Yes |
| OpenSMTPD | C | ISC | 7.0+ | Yes |

### Multi-tenant assessment

**Postfix** — No native multi-tenancy. Requires per-tenant domain/vhost
configuration generated by config compiler. Queue partitioning by tenant
(`/var/spool/postfix/incoming/tenant-{uuid}/`). Virtual mailbox domains map
multiple tenant domains to local delivery. Rate limiting per domain via
`smtpd_client_connection_rate_limit` and `smtpd_client_message_rate_limit`.
**Multi-tenancy effort: moderate.** Config compiler drives all tenant config.

**Stalwart SMTP** — Native multi-tenancy. Built-in tenant model with per-tenant
policies, quotas, domains, and users. SMTP is part of the Stalwart unified
server; all tenant data is isolated by the server's internal model.
**Multi-tenancy effort: trivial** (it's native).

**Haraka** — Plugin-based, so multi-tenancy must be built in custom plugins.
`rcpt_to.routes` plugin can route to different backends based on domain. Redis
plugin for state. `auth/` plugins support various auth backends.
**Multi-tenancy effort: high** (custom plugin required for tenant isolation).

**OpenSMTPD** — Table-based configuration, supports virtual hosts but no
built-in tenant model. Multi-tenancy requires manual configuration per
virtual host. Limited plugin ecosystem.
**Multi-tenancy effort: moderate-high.**

### Detailed comparison

| Criterion | Postfix | Stalwart SMTP | Haraka | OpenSMTPD |
|-----------|---------|---------------|--------|-----------|
| Operational simplicity | 9/10 | 8/10 | 6/10 | 7/10 |
| Protocol coverage | 9/10 | 8/10 | 7/10 | 6/10 |
| Extensibility | 6/10 | 8/10 | 9/10 | 5/10 |
| Backup story | 8/10 | 9/10 | 5/10 | 7/10 |
| Search quality | 3/10 (none) | 7/10 (native) | 3/10 (none) | 3/10 (none) |
| Groupware maturity | 3/10 (SMTP only) | 9/10 (full stack) | 3/10 (SMTP only) | 3/10 (SMTP only) |
| Abuse integration | 8/10 (Rspamd milter) | 7/10 (internal filter) | 6/10 (plugin) | 5/10 |
| **Multi-tenancy** | **6/10** (workaround) | **9/10** (native) | **3/10** (plugins needed) | **5/10** |
| Maturity/stability | 10/10 | 6/10 | 7/10 | 7/10 |
| Ecosystem/middleware | 10/10 | 5/10 | 6/10 | 6/10 |

### Recommendation

- **Track A (integrated):** Stalwart SMTP is the default — multi-tenancy is
  native, protocol support is comprehensive, and it's the same codebase as the
  mailbox/backend.
- **Track B (composable Unix):** Postfix is the default — battle-tested, config
  compiler can drive tenant configs, Rspamd milter integration is proven.

## 2. Mailbox Backend / IMAP

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| Dovecot | C | MIT/LGPL | 2.3+ | 25+ years |
| Stalwart IMAP | Rust | AGPL 3.0 | 0.22+ | Yes |
| Cyrus IMAP | C | Various | 3.8+ | 25+ years |
| Mailcow (Dovecot) | Docker | GPL | 2024+ | Yes |

### Multi-tenant assessment

**Dovecot** — No native multi-tenancy. Supports `mail_location` with per-user
directory structure (`/var/mail/%d/%n/` for domain/user) but tenant grouping
requires external management. Namespace partitioning is per-user, not per-tenant.
Shared mailboxes can be cross-tenant if not carefully scopped. Auth via SQL/
LDAP with tenant_id in the user record. Sieve scripts stored per-user.
**Multi-tenancy effort: high.** Requires config compiler to generate per-tenant
mail_location paths, namespace prefixes, and auth queries. Critical need for
tenant-scoped auth queries to prevent cross-tenant login.

**Stalwart IMAP** — Native multi-tenancy. Tenant-aware from the ground up.
IMAP connections are bound to the tenant. All mailboxes, folders, messages,
and resources are scoped to tenant_id internally. Shared resources require
explicit cross-tenant ACLs.
**Multi-tenancy effort: trivial.**

**Cyrus IMAP** — No native multi-tenancy. Uses a flat namespace with ACLs.
Multi-tenancy requires manual namespace separation or careful ACL design.
Mail storage is Cyrus-specific format. Good for shared mailboxes but weak
for SaaS isolation.
**Multi-tenancy effort: very high.** Not recommended for multi-tenant SaaS.

### Detailed comparison

| Criterion | Dovecot | Stalwart IMAP | Cyrus |
|-----------|---------|---------------|-------|
| Operational simplicity | 9/10 | 7/10 | 5/10 |
| Protocol coverage | 10/10 | 8/10 | 9/10 |
| Extensibility | 9/10 | 7/10 | 7/10 |
| Backup story | 8/10 | 9/10 (SQL + blob) | 6/10 (Cyrus format) |
| Search quality | 7/10 (Xapian) | 8/10 (native + external) | 3/10 (none) |
| Groupware maturity | 9/10 | 8/10 (built-in) | 9/10 |
| Abuse integration | 8/10 (Sieve) | 7/10 (internal) | 7/10 |
| **Multi-tenancy** | **5/10** | **9/10** | **4/10** |
| Maturity/stability | 10/10 | 6/10 | 9/10 |
| Ecosystem/middleware | 10/10 | 5/10 | 7/10 |

### Recommendation

- **Track A:** Stalwart IMAP — native tenant isolation, same codebase as SMTP.
- **Track B:** Dovecot — proven, extensive ecosystem, requires careful config
  compiler tenant scoping.

## 3. Abuse Engine

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| Rspamd | Rust/C | BSD 3-clause | 4.0+ | Yes |
| SpamAssassin | Perl | ASF | 4.0+ | Yes |
| amavisd-new | Perl | GPL | 2.13+ | Yes |
| MailGuard | Perl | AGPL | N/A | No (abandoned) |

### Multi-tenant assessment

**Rspamd** — No native multi-tenancy in the core, but tenant separation is
achievable through: (1) per-tenant maps for rules/allowlists, (2) Redis state
with tenant key prefixing, (3) per-tenant rate limiting via Redis, (4) per-
tenant Bayes training. Config compiler generates tenant-scoped configuration.
Rspamd's Lua filters can reference tenant context from HTTP headers or auth
context. Rspamd 4.x added multi-instance support useful for tenant separation.
**Multi-tenancy effort: moderate.** Config compiler handles config generation;
Redis key prefixing handles state isolation; per-tenant maps handle rules.

**SpamAssassin** — No tenant awareness. All scoring is global. Bayes DB is
shared. No way to isolate per-tenant rules or scoring.
**Multi-tenancy effort: very high.** Effectively not usable for multi-tenant.

**amavisd-new** — No tenant awareness. Processes mail globally. Can route to
different backends but scoring is shared.
**Multi-tenancy effort: very high.**

### Recommendation

**Rspamd** — Only viable option. Its Redis-based architecture and Lua extensibility
make tenant scoping achievable. Per-tenant maps for rules, shared Redis with
key prefixing for state, and config-driven configuration.

## 4. Malware Scanning

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| ClamAV | C | GPL | 1.4+ | Yes |
| Sophos AV CLI | C | Proprietary | N/A | Yes |
| OpenDLP | Python | Apache 2.0 | 3.0+ | Yes |
| McAfee Engine | C | Proprietary | N/A | Yes |

### Multi-tenant assessment

**ClamAV** — Stateless daemon. No tenant awareness needed — it scans individual
files and returns pass/fail. Multi-tenancy is implicit: tenant isolation happens
at the scan request level (which tenant's message is being scanned). ClamAV
does not store state, so no tenant data leakage is possible.
**Multi-tenancy effort: none.** State scanning is inherently safe.

**Sophos/McAfee** — Same as ClamAV: stateless scanning.

### Recommendation

**ClamAV** — Free, mature, integrates with Rspamd natively. No multi-tenancy
concerns since it's stateless.

## 5. Policy State / Cache

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| Redis | C | BSD 3-clause | 7.0+ | Yes |
| Valkey | C | BSD 4-clause | 8.0+ | Yes |
| Memcached | C | BSD 3-clause | 1.6+ | Yes |

### Multi-tenant assessment

**Redis** — No native multi-tenancy, but key prefix convention (`tenant:{id}:...`)
provides full isolation. Redis ACL (Redis 6+) can limit per-connection access
to specific key patterns — config compiler generates ACL per tenant. Redis
Cluster adds data distribution.
**Multi-tenancy effort: moderate.** Key prefix convention + ACL for strong
isolation.

**Valkey** — Redis fork donated by Oracle. Same capabilities. No advantage
over Redis for multi-tenancy.

**Memcached** — No ACL, no key prefixing enforcement, no persistence. Worse
for multi-tenancy. Not recommended.

### Recommendation

**Redis** — The standard. Redis ACL for per-tenant key access control.

## 6. Calendar / Contacts (DAV/JMAP)

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| Stalwart DAV | Rust | AGPL 3.0 | 0.22+ | Yes |
| Radicale | Python | GPL 3.0 | 3.0+ | Yes |
| SabreDAV | PHP | MIT | 4.0+ | Yes |
| DaviCal | PHP | GPL 2.0 | 1.4+ | Yes |

### Multi-tenant assessment

**Stalwart DAV** — Native multi-tenancy. CalDAV/CardDAV/JMAP Calendar/JMAP
Contacts are built-in. Tenant-scoped from the ground up. Shared resources
require explicit ACLs.
**Multi-tenancy effort: trivial.**

**Radicale** — No native multi-tenancy. Uses a flat filesystem store.
Multi-tenancy requires filesystem-level directory isolation per tenant or
custom authentication plugin. No ACL support for resource sharing.
**Multi-tenancy effort: very high.**

**SabreDAV** — Framework, not a product. Multi-tenancy must be built into
the implementing application. No built-in tenant model.
**Multi-tenancy effort: very high** (must build the product plane around it).

**DaviCal** — PostgreSQL-backed, so tenant isolation is possible via RLS.
PostgreSQL-backed, so tenant isolation is possible via RLS. Supports sharing
but no tenant model.
**Multi-tenancy effort: moderate** (RLS possible since it's PostgreSQL-based).

### Recommendation

- **Track A:** Stalwart DAV — native multi-tenancy, same codebase.
- **Track B:** Radicale or custom DAV layer over Stalwart/SabreDAV. SabreDAV
  as a framework is more flexible but requires building tenant-aware auth.

## 8. Search Engine

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| OpenSearch | Java | Apache 2.0 | 3.0+ | Yes |
| Tantivy | Rust | MIT | 0.20+ | Yes |
| Xapian | C++ | GPL | 1.4+ | Yes |
| Meilisearch | Rust | MIT | 1.0+ | Yes |

### Multi-tenant assessment

**OpenSearch** — No native multi-tenancy, but index-level isolation works well:
each tenant gets a dedicated index (`tenant-{uuid}-2026.*`). Queries always
include `tenant_id` filter. Multi-index queries for admin search across tenants.
**Multi-tenancy effort: moderate.** Index-per-tenant pattern is standard.

**Tantivy** — Rust-based, embedded library (not a server). Can be integrated
directly into the product code. Tenant isolation is application-level: index
files stored in tenant-scoped directories. No network boundary, so no
multi-tenancy concerns beyond application logic.
**Multi-tenancy effort: low** (embedded, tenant-scoped filesystem).

**Xapian** — Embedded library (like Tantivy). PostgreSQL-backed query capability.
Tenant isolation through database scoping.
**Multi-tenancy effort: low** (database-backed).

**Meilisearch** — No native multi-tenancy. Uses API keys and index isolation.
Tenant isolation via separate indexes per tenant. No multi-index search in
free tier.
**Multi-tenancy effort: moderate** (index-per-tenant, limited multi-tenant search).

### Recommendation

- **Tantivy** — Rust-native, embedded, no separate service to manage. Perfect
  for product-plane integration where tenant context is always known.
- **OpenSearch** — If external search server is needed (cross-tenant admin
  search, full-text admin queries). Index-per-tenant pattern.

## 9. Blob / Object Storage

### Candidates

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|
| MinIO | Go | AGPL | 2025+ | Yes |
| Ceph (RGW) | C++ | LGPL | 18.0+ | Yes |
| Cloud S3 | N/A | Commercial | N/A | Yes |
| Local filesystem | N/A | N/A | N/A | N/A |

### Multi-tenant assessment

**MinIO** — S3-compatible. No native multi-tenancy, but prefix-based isolation
is the standard pattern. IAM policies per prefix allow fine-grained access.
Cross-cluster replication available.
**Multi-tenancy effort: low** (prefix + IAM, standard S3 pattern).

**Ceph RGW** — Supports tenant-aware ACLs and bucket policies. More complex
deployment but can enforce per-tenant storage quotas at the storage layer.
**Multi-tenancy effort: moderate** (more complex, more features).

**Cloud S3** — No native multi-tenancy but prefix + IAM policy pattern works.
No additional cost for multi-tenancy.
**Multi-tenancy effort: none** (uses cloud provider features).

### Recommendation

**MinIO for self-hosted, Cloud S3 for production.** MinIO gives full control
and is the simplest to deploy in K8s (StatefulSet). Cloud S3 is simpler for
production (no infrastructure to manage).

| Option | Language | License | Version | Active |
|--------|----------|---------|---------|--------|

| Zitadel | Go | AGPL 3.0 | 2.50+ | Yes |
| external IdP | N/A | N/A | N/A | N/A |

### Multi-tenant assessment

**Keycloak** — Native multi-tenancy via "realms." Each tenant maps to a Keycloak
realm. Cross-realm federation possible. Well-established SaaS pattern.
**Multi-tenancy effort: low** (one realm per tenant, standard pattern).

**Authentik** — Native multi-tenancy via "tenants." Designed for SaaS from

**Multi-tenancy effort: low** (native SaaS design).

**Zitadel** — Native multi-tenancy via "organizations." Each tenant is an
organization. Built as a B2B identity platform.
**Multi-tenancy effort: low** (native SaaS design).

### Recommendation

- **Authentik** — Best native SaaS/multi-tenant experience. Designed for this
  use case from the ground up.
- **Keycloak** — More mature ecosystem, more integrations. Realm-per-tenant
  pattern is well-established.

## Key findings

1. **Stalwart (Track A) wins on multi-tenancy** across every component. The
   entire stack shares a native tenant model, so tenant isolation is built-in
   rather than bolted on. This is the strongest argument for Track A.

2. **Track B (Dovecot/Postfix) requires significantly more work** to achieve
   the same level of multi-tenant isolation. The config compiler must drive
   tenant-scoped configuration for every component, and tenant leakage risks
   are higher (wrong query, wrong path, wrong namespace).

3. **Rspamd is the only viable abuse engine** for multi-tenant deployment.
   SpamAssassin and amavisd-new have no tenant awareness.

4. **ClamAV is stateless** — no multi-tenant concerns at all.

5. **Search: Tantivy** (embedded, Rust) vs **OpenSearch** (external server).
   Tantivy is simpler (no service to manage) but limited for cross-tenant
   admin searches. OpenSearch supports multi-index queries but adds a service.

6. **OIDC: Authentik** is the most multi-tenant-friendly (designed for SaaS).
   Keycloak has the largest ecosystem but realm-per-tenant is a workaround.

7. **The config compiler must be the single source of truth** for tenant
   isolation regardless of Track A or Track B. It generates per-tenant
   configuration for every component.

## Decision gate criteria

When choosing between Track A and Track B, evaluate:

1. **Team familiarity** — Does the team know Rust (Stalwart) or C/Linux
   mail (Postfix/Dovecot) better?
2. **Multi-tenant complexity budget** — Track A saves ~6 months of tenant
   isolation work across the stack.
3. **Protocol requirements** — If JMAP is required, Stalwart is the only
   FOSS option with native JMAP.
4. **Operational overhead** — Track B has more moving parts (Postfix + Dovecot
   + CalDAV + separate DAV service vs Stalwart integrated stack).
5. **Maturity trade-off** — Track B components are 25+ years old; Stalwart
   is new (2023+).
